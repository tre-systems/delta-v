# Accessibility (DOM) - audit guide

Delta-V uses a Canvas game board. Full non-visual play would require a parallel interface and is not currently in scope. This guide focuses on the accessible parts of the product we do control: the menu, lobby, overlays, heads-up display chrome, fleet builder, chat, and buttons.

## Scope

The surfaces covered by this audit are the client user-interface modules and the DOM helper module, along with any overlay and lobby controls rendered outside the Canvas.

What is out of scope for now is the pointer-first ship and hex interaction on the Canvas board, and full keyboard-only tactical play parity.

## Audit cadence

Run this checklist before release candidates, after major user-interface refactors, and after introducing new modal, form, or heads-up display interaction patterns.

## Automated checks

Run both automated paths when possible. First, run a Lighthouse Accessibility audit on the main page. Second, run the end-to-end accessibility test suite, which covers the menu, waiting lobby, and heads-up display and help overlay DOM. That suite fails on serious or critical violations and excludes the Canvas and color-contrast checks, which are tracked manually in the checklist.

Pass criteria are that there are no critical or serious accessibility violations on the audited views, and that any new violations are either fixed or explicitly documented in the project backlog.

## Manual checklist (pass/fail)

- Keyboard order: Tab order reaches the Create and Join actions and primary controls in the expected sequence.
- Activation semantics: Enter and Space activate controls; native button elements are preferred over clickable divs.
- Form naming: Room code and chat inputs have visible labels or a correct ARIA label attribute.
- Focus visibility: The keyboard focus ring or outline is visible on all actionable elements.
- Modal behavior: Focus is trapped while a modal is open and restored to the trigger element on close.
- Contrast: Heads-up display and overlay text is readable on translucent backgrounds, targeting WCAG AA compliance where feasible; follow the contrast and readability spot-check steps in the manual test plan for glassy panels such as the help overlay and game-over overlay.
- Status messaging: Important toasts and errors use the appropriate ARIA role — either "status" or "alert" — where a signal is needed without excessive noise.

## Canvas limitation note

Document Canvas constraints explicitly in pull requests and issues when relevant. Do not create placeholder keyboard-support tasks without a concrete product decision and implementation plan.

## Canvas role-application and board summary

The tactical map stays with the ARIA role "application" and a long ARIA label so assistive tech exposes it as a single complex control rather than browse-mode noise. Full non-visual play on the hex grid is still out of scope.

To partially offset the loss of browse mode, the hidden board-summary live region — configured with a polite ARIA-live setting and a status role — is updated from the heads-up-display chrome with high-level board context including turn, phase, and objective text. That documents the current product choice: keep the application role on the canvas and push lightweight state narration to a dedicated live region instead of switching the canvas to an image role.

## Baseline record template

When running an audit, add a short note to the pull request or backlog using the following structure: the date, the surfaces checked, the automated results, a manual checklist summary, any new issues found, and links to any follow-up tickets or backlog entries.

## Latest audit

The most recent audit was conducted on April 24, 2026. The surfaces checked were the automated end-to-end accessibility suite with eight out of eight axe checks passing, the menu, the in-game heads-up display, the help overlay, the game-over overlay, and the reconnect overlay at a 375 by 812 pixel viewport, the focus-visible CSS inventory, coverage of the reduced-motion and high-contrast user preferences, the live-region inventory, and heads-up-display scale discoverability.

The automated end-to-end accessibility command passed all eight axe checks on the menu, menu contrast, waiting lobby, in-game heads-up display and help overlay, keyboard activation of primary navigation, help-overlay focus move and restore, fleet-building screen, and desktop log panel.

Manual checklist summary. Keyboard order: primary flows pass; the help, game-over, and reconnect overlays implement Tab focus traps. Activation semantics: native buttons and radios with role attributes; Enter and Space work. Form naming: callsign, room-code, and chat inputs have ARIA labels or visible labels. Focus visibility: a global focus-visible ring is applied at two pixels solid accent with a two-pixel offset. Modal behavior: the help overlay, the game-over overlay, and the reconnect overlay all have modal-dialog semantics and Tab focus traps — a change from the previous audit where only the help overlay trapped focus. Contrast: the more-contrast media query drops backdrops on the heads-up display, overlays, phase alert, and menu; translucent heads-up-display text still needs per-device spot-checking on real hardware. Status messaging: fourteen ARIA-live regions are present in the main HTML entry — including the hidden board summary, callsign status, game code, toast container, phase alert, log entries, log latest bar, opponent-disconnect overlay, new-version banner, menu offline banner, official-bot offer, chat character counter, log status bar, and logistics container. Coverage is comprehensive.

New issues found in this audit. The reduced-motion preference disables only five CSS selectors, but twelve keyframe blocks and thirty-two animation or transition rules otherwise fire; the Canvas animation layer ignores the setting entirely, with no match-media reduced-motion check anywhere in the client. Also, the heads-up-display scale is keyboard-only, adjustable with the plus and underscore keys, with no touch-reachable UI and no visible discoverability — touch users who most need larger text cannot adjust it.

These findings were filed as a single low-friction accessibility polish entry at priority two in the backlog. Full keyboard tactical play on the canvas board remains explicitly out of scope.

### Automated code scan (2026-04-03)

This scan was conducted on April 3, 2026. It covered all client user-interface view files, the main HTML entry point, and the DOM helper module. The method was a grep-based scan for clickable non-interactive elements missing a role attribute, a tab-index, an ARIA label, or keyboard handlers.

Three issues were found and fixed. First, fleet cart chips in the fleet-building view were clickable div elements that lacked a button role, a tab-index, an ARIA label, and keyboard activation — all of which were added. Second, ship list entries in the ship-list view already had a button role and an ARIA label but were missing a tab-index and an Enter and Space keyboard handler — both were added. Third, the log latest bar in the game-log view was a clickable div used to expand the game log, and it lacked a button role, a tab-index, and a keyboard handler — all three were added.

No missing alternative text on images, no missing ARIA labels on icon buttons, and no form inputs without labels were found. The TypeScript compiler check and the full Vitest suite both pass.
