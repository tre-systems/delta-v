# Accessibility (DOM) - audit guide

Delta-V uses a Canvas game board. Full non-visual play would require a parallel interface and is not currently in scope. This guide focuses on the accessible parts of the product we do control: the menu, lobby, overlays, heads-up display chrome, fleet builder, chat, and buttons.

## Scope

The surfaces covered by this audit are the client user-interface modules and the DOM helper module, along with any overlay and lobby controls rendered outside the Canvas.

What is out of scope for now is the pointer-first ship and hex interaction on the Canvas board, and full keyboard-only tactical play parity.

## Audit cadence

Run this checklist before release candidates, after major user-interface refactors, and after introducing new modal, form, or heads-up display interaction patterns.

## Automated checks

Run both automated paths when possible. First, run a Lighthouse Accessibility audit on the main page. Second, run the end-to-end accessibility test suite, which covers the menu, waiting lobby, and heads-up display and help overlay DOM. That suite fails on serious or critical violations and excludes the Canvas and color-contrast checks, which are tracked manually in the checklist.

Pass criteria are that there are no critical or serious accessibility violations on the audited views, and that any new violations are either fixed or explicitly documented in the project backlog.

## Manual checklist (pass/fail)

- **Keyboard order:** Tab order reaches the Create and Join actions and primary controls in the expected sequence.
- **Activation semantics:** Enter and Space activate controls; native button elements are preferred over clickable divs.
- **Form naming:** Room code and chat inputs have visible labels or a correct ARIA label attribute.
- **Focus visibility:** The keyboard focus ring or outline is visible on all actionable elements.
- **Modal behavior:** Focus is trapped while a modal is open and restored to the trigger element on close.
- **Contrast:** Heads-up display and overlay text is readable on translucent backgrounds, targeting WCAG AA compliance where feasible.
- **Status messaging:** Important toasts and errors use the appropriate ARIA role — either "status" or "alert" — where a signal is needed without excessive noise.

## Canvas limitation note

Document Canvas constraints explicitly in pull requests and issues when relevant. Do not create placeholder keyboard-support tasks without a concrete product decision and implementation plan.

## Baseline record template

When running an audit, add a short note to the pull request or backlog using the following structure: the date, the surfaces checked, the automated results, a manual checklist summary, any new issues found, and links to any follow-up tickets or backlog entries.

## Latest audit

The most recent audit was conducted on April 3, 2026. The surfaces checked were the main menu, waiting lobby, heads-up display chrome, help overlay, chat and log DOM, and the toast, reconnect, and game-over overlays.

The automated end-to-end accessibility suite passed all five tests. The manual checklist results were as follows. Keyboard order passed for menu primary actions and the help overlay open and close flow. Activation semantics passed on all audited controls, with native buttons remaining the default. Form naming was fixed for the room-code and chat inputs by adding explicit ARIA labels. Focus visibility had no code regression found, though it remains worth checking on real devices and browsers during release passes. Modal behavior improved significantly: the help overlay now traps focus and restores it on close, and the reconnect, game-over, and help overlays now expose dialog semantics. Contrast remains a human visual check, especially on translucent heads-up display surfaces. Status messaging was addressed by giving toasts, the latest-log bar, the reconnect banner, and the phase alert live-region semantics.

No serious or critical automated violations were found after these fixes. Future visual and manual accessibility issues should be added to the project backlog if later passes find concrete regressions.

### Automated code scan (2026-04-03)

This scan was also conducted on April 3, 2026. It covered all client user-interface view files, the main HTML entry point, and the DOM helper module. The method was a grep-based scan for clickable non-interactive elements missing a role attribute, a tab-index, an ARIA label, or keyboard handlers.

Three issues were found and fixed. First, fleet cart chips in the fleet-building view were clickable div elements that lacked a button role, a tab-index, an ARIA label, and keyboard activation — all of which were added. Second, ship list entries in the ship-list view already had a button role and an ARIA label but were missing a tab-index and an Enter and Space keyboard handler — both were added. Third, the log latest bar in the game-log view was a clickable div used to expand the game log, and it lacked a button role, a tab-index, and a keyboard handler — all three were added.

No missing alternative text on images, no missing ARIA labels on icon buttons, and no form inputs without labels were found. The TypeScript compiler check and the full Vitest suite both pass.

## Follow-ups

Track concrete fixes in the project backlog. Prefer specific, testable items over broad "improve accessibility" entries.
