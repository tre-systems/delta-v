# Privacy — technical summary

Describes what the Delta-V stack stores. Not a privacy policy, legal review, or jurisdiction-specific compliance advice.

## Scope

This document covers telemetry and error collection, server-side diagnostics, and match-archive storage. It does not define legal basis, consent language, or user-facing policy wording.

## Client data flow

Each client is assigned a random anonymous identifier — a UUID stored in the browser's local storage under the Delta-V anonymous-identifier key — that is attached to all client reporting payloads.

The lobby separately stores a player profile — player key, username, and last-updated timestamp — under the Delta-V player-profile key so a returning browser keeps the same callsign and matchmaking identity. The lobby exposes a "Forget my callsign" control that removes this local profile and clears cached room tokens on the current device. The control does not rotate the anonymous telemetry UUID, so post-reset telemetry still links to pre-reset rows via the anonymous identifier. This is tracked in the backlog.

Room-scoped player tokens are cached in local storage under the Delta-V tokens key for reconnect and join convenience. The cache is pruned to the most recent eight entries and drops anything older than twenty-four hours.

When an error occurs, the reporting helper sends the error itself, any context the caller supplies, the page URL, and the browser user-agent string. When an event is tracked, it sends an arbitrary event name along with any associated properties. A firm requirement at every call site is that secrets, credentials, and direct personal data must never appear in telemetry or error properties.

## Server storage

The server exposes two endpoints for receiving telemetry and error reports, each accepting JSON payloads up to four kilobytes. Before any data is written to the database, the client's IP address is transformed into a hashed value. The hash is computed as SHA-256 of a secret salt, a colon separator, and the IP address, truncated to sixteen hexadecimal characters; the raw IP address is never written. A dedicated production IP-hash-salt environment variable can provide the salt; when unset in non-production, the existing agent-token secret is used so hashes remain resistant to forward lookup. In production the server fails closed if the IP-hash salt is not configured. Rotating the active salt makes future rows unlinkable from historic rows through the application hash. Historic hashes are not retroactively salted.

Events are stored in a database events table that holds a timestamp, the anonymous client identifier, the event name, the event properties, the hashed IP, and the user-agent string — see the create-events migration for the schema.

Turning to server-side diagnostics: events generated by the game engine — such as engine errors, state-projection mismatches, abandoned games, and other lifecycle events — are written with the fixed server sentinel "server" in place of the hashed IP field. These diagnostic payloads may include stack traces. Stack-trace scrubbing is a priority-three follow-up in the backlog; the likelihood of user-typed-string exposure today is low, but worth bounding when the error-reporting path is next touched.

## Match and gameplay data

Match metadata is stored in the main database's match-archive table. Completed match archives are stored in object storage under the matches-slash-game-id-dot-JSON key pattern. Chat messages are transmitted over WebSocket only and are not written to the database through the default telemetry or error path.

## Leaderboard data

Humans claim a leaderboard username through the claim-name endpoint, and agents claim one by passing a claim object with a username when requesting an agent token. Either path binds a client-held player key to a user-chosen username in the player table. The "is agent" flag is set only from the verified agent path, never from the player-key prefix alone.

A second table, called match rating, stores one row per rated match. Each row records the game identifier, both participating player keys, the winner, and Glicko-2 rating snapshots from before and after the match. Rows are keyed on game identifier, so a replay or retry of the same match is an idempotent no-op via insert-or-ignore.

The public leaderboard API — both the main listing and the per-player rank lookup — exposes only the username, the Glicko-2 rating triple, games played, distinct opponents, the timestamp of the last rated match, and the "is agent" flag. Player keys are never returned in public responses. The public matches API does not expose usernames; the public match log includes only match metadata plus replay-identifying room and game identifiers.

## Operational note

Access to logs, database query surfaces, and match archives is restricted to trusted maintainers. The product ships a short in-app operational disclosure describing anonymous diagnostics and completed-match archives. That disclosure is not a substitute for legal review.
