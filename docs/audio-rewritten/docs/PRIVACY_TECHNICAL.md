# Privacy — technical summary

Describes what the Delta-V stack stores. Not a privacy policy, legal review, or jurisdiction-specific compliance advice.

## Scope

This document covers telemetry and error collection, server-side diagnostics, and match-archive storage. It does not define legal basis, consent language, or user-facing policy wording.

## Client data flow

Each client is assigned a random anonymous identifier — stored in the browser's local storage — that is attached to all client reporting payloads. When an error occurs, the reporting helper sends the error itself, any context the caller supplies, the page URL, and the browser user-agent string. When an event is tracked, it sends an arbitrary event name along with any associated properties. A firm requirement at every call site is that secrets, credentials, and direct personal data must never appear in telemetry or error properties.

## Server storage

The server exposes two endpoints for receiving telemetry and error reports, each accepting JSON payloads up to four kilobytes. Before any data is written to the database, the client's IP address is transformed into a hashed value using SHA-256 and then truncated; the raw IP address is never written.

Events are stored in a database table that holds a timestamp, the anonymous client identifier, the event name, the event properties, the hashed IP, and the user-agent string.

Turning to server-side diagnostics: events generated by the game engine — such as engine errors, state-projection mismatches, abandoned games, and other lifecycle events — are written with a fixed server sentinel in place of the hashed IP field. These diagnostic payloads may include stack traces.

## Match and gameplay data

Match metadata is stored in the main database. Completed match archives are stored in object storage, keyed by game identifier. Chat messages are transmitted over WebSocket only and are not written to the database through the normal telemetry or error path.

## Leaderboard data

Humans claim a leaderboard username through the claim-name endpoint, and agents claim one as part of the agent-token mint flow. Either path binds a client-held player key to a user-chosen username in the player table. The "is agent" flag is set only from the verified agent path, never from the player-key prefix alone.

A second table, called match rating, stores one row per rated match. Each row records the game identifier, both participating player keys, the winner, and Glicko-2 rating snapshots from before and after the match. Rows are keyed on game identifier, so a replay or retry of the same match is an idempotent no-op.

The public leaderboard API — both the main listing and the per-player rank lookup — exposes only the username, the Glicko-2 rating triple, games played, distinct opponents, the timestamp of the last rated match, and the "is agent" flag. Player keys are never returned in public responses.

## Operational note

Access to logs, database query surfaces, and match archives is restricted to trusted maintainers. The product ships a short in-app operational disclosure describing anonymous diagnostics and completed-match archives. That disclosure is not a substitute for legal review.
