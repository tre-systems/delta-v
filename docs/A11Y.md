# Accessibility (DOM) — review checklist

The **game board is Canvas**; full non-visual play is out of scope without a parallel UI. This checklist targets **menus, lobby, HUD chrome, fleet builder, chat field, and buttons** (`src/client/ui/`, `src/client/dom.ts`).

## Automated (when convenient)

- [ ] Lighthouse **Accessibility** on `/` (or Playwright + axe-core).
- [ ] No critical axe violations on lobby and in-game overlay DOM.

## Manual

- [ ] **Tab order** reaches Create / Join / primary actions before Canvas traps focus.
- [ ] **Enter / Space** activate buttons (native `<button>` preferred over `<div>` click handlers).
- [ ] **Form fields** (room code, chat) have associated **labels** or `aria-label`.
- [ ] **Focus visible** on interactive elements (keyboard users can see focus).
- [ ] **Modal / overlay** traps focus or returns focus on close where applicable.
- [ ] **Color contrast** acceptable for HUD text on glass-style backgrounds (WCAG AA where feasible).
- [ ] **Toasts / errors**: consider `role="status"` or `role="alert"` for important messages (evaluate noise vs benefit).

## Canvas limitations (document, don’t “fix” blindly)

- Ship selection and hex interaction are pointer-first; document **keyboard alternative** only if product requires it (large effort).
